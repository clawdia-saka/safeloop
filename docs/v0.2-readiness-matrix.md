# SafeLoop v0.2 readiness matrix

This matrix is the v0.2 completion snapshot for SafeLoop after the public demo, operator packet, explain UX, and compensation adapter example work. It is intentionally a documentation and audit artifact only; it does not add runtime features.

## Boundary statement

SafeLoop guarantees exact rollback only for covered local file changes that are captured in its local artifacts. Actions outside the local repo—GitHub, messaging, email, webhook delivery, hosted systems, or other remote services—must be handled through compensation and manual review. They are never claimed as exact rollback.

Statuses used in this document:

- `complete`: implemented and covered by docs/tests/examples in this repo.
- `partial`: present, but with explicit limitations or missing product hardening.
- `planned`: intended future work, not complete in v0.2.
- `out_of_scope`: intentionally not part of v0.2.

## Reading guide

- If a capability is `complete`, it is complete for the local v0.2 scope and backed by at least one file, test, example, or doc.
- If a capability is `partial`, it is usable in a local/demo context but still has hardening gaps.
- If a capability is `planned`, it is future work and must not be marketed as shipped.
- If a capability is `out_of_scope`, SafeLoop should explicitly say it is not part of v0.2 rather than imply support.

The most important product boundary remains: local file rollback can be exact when covered by SafeLoop artifacts; actions outside the local repo require compensation and manual review, not exact rollback.
